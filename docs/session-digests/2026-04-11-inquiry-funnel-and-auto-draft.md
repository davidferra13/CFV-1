### 3. Auto-Draft Reply (first-response email generated on inquiry creation)

- `lib/messages/internal.ts` — new non-auth `insertDraftMessageInternal()` for background use
  - Idempotent: skips if outbound draft already exists for the inquiry
- `lib/ai/reactive/handlers.ts` — `handleInquiryCreated` now:
  1. Scores the lead (unchanged)
  2. Calls `generateFirstResponseDraft()` via Ollama (local, PII-safe)
  3. Saves result to `messages` table as `status: 'draft', direction: 'outbound'`
  - Non-blocking: if Ollama is offline the OllamaOfflineError propagates; other errors are logged and swallowed
- Draft prompt constraints: warm/brief/personal, one qualifying question, no pricing, no em dashes, <150 words
- `InquiryResponseComposer` updated with `pendingDraft` prop:
  - Pre-populates subject, body, and savedMessageId from the DB record
  - Green "Draft ready" badge in header
  - Badge clears on Regenerate or Discard
  - Discard deletes the DB record non-blocking (UI updates immediately)
  - `isAutoGenerated` flag tracks badge state separately from draft content
- `app/(chef)/inquiries/[id]/page.tsx`:
  - Extracts first outbound draft from already-fetched messages (zero extra DB queries)
  - Passes to composer as `pendingDraft`
  - Filters `status: 'draft'` messages OUT of `MessageThread` (no duplicates in comm log)

## Key Architecture Decisions

- Draft lives in `messages` table with `status: 'draft'` - same table as sent messages
- Background handler uses `createAdminClient()` not session-auth (no chef session in queue worker)
- Draft idempotency guard prevents double-generation if handler retries
- Page reads draft from the already-fetched `messages` array - no second query
- Discard is fire-and-forget on the DB side; UI is never blocked
