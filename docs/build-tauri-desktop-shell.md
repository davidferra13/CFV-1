# ChefFlow Desktop App — Tauri Shell

## What This Is

ChefFlow now ships as both a website and a native desktop app. The desktop app is a **Tauri shell** — a thin native wrapper that loads the real ChefFlow web app inside a built-in Windows WebView2 window.

You get:

- **System tray** — ChefFlow lives in your notification area, always one click away
- **Close to tray** — clicking X hides the window; the app never actually quits
- **Right-click tray menu** — Open, New Event, New Inquiry, Quit
- **Auto-start on login** — toggle in Settings → Desktop App
- **Native desktop notifications** — for inquiries, messages, payments, upcoming events
- **~10MB installer** — lightweight because it uses the browser engine already built into Windows (WebView2)

The website continues to work perfectly in any browser. The desktop app is an additional delivery method.

## Architecture

```
┌──────────────────────────────────────────┐
│  Tauri Desktop Shell                     │
│  ┌────────────────────────────────────┐  │
│  │  WebView2 (Windows built-in)       │  │
│  │  loads http://localhost:3100       │  │
│  │  (or your production URL)          │  │
│  └────────────────────────────────────┘  │
│  + System Tray (Rust)                    │
│  + Auto-start on Windows login           │
│  + Native desktop notifications          │
│  + "Close to tray" window behavior       │
└──────────────────────────────────────────┘
           ↑ same backend ↑
┌──────────────────────────────────────────┐
│  Next.js App — fully unchanged           │
│  Supabase + Stripe — unchanged           │
└──────────────────────────────────────────┘
```

Key insight: the Tauri shell contains **zero business logic**. It just wraps the URL. All auth, data, and features live in the Next.js app. The chef signs in once in the desktop app (stored in the WebView2 profile) and stays signed in.

## How It Works

### Close to Tray

In `src/main.rs`, the `on_window_event` handler intercepts `CloseRequested`:

```rust
.on_window_event(|window, event| {
    if let tauri::WindowEvent::CloseRequested { api, .. } = event {
        window.hide().unwrap();   // hide window
        api.prevent_close();      // don't actually close
    }
})
```

The process stays alive. The tray icon stays visible. Right-click → Quit is the only way to fully exit.

### System Tray

In `src/tray.rs`, `TrayIconBuilder` creates the tray entry with a context menu. Left-click shows/focuses the window. Menu items:

- **Open ChefFlow** → `window.show()` + `window.set_focus()`
- **New Event** → show window + `window.eval("window.location.href = '/events/new'")`
- **New Inquiry** → show window + navigate to `/inquiries/new`
- **Quit ChefFlow** → `app.exit(0)`

### Auto-Start

The `tauri-plugin-autostart` plugin writes a Windows registry entry (or macOS login item). The Settings page (`DesktopAppSettings` component) calls the plugin's JS API bridge to toggle it. When not running in Tauri, the component shows a download prompt instead.

### Desktop Notifications

`lib/notifications/desktop-notify.ts` exposes `sendDesktopNotification({ title, body })`. It:

1. Checks for `window.__TAURI_INTERNALS__` to detect Tauri
2. Requests notification permission if needed
3. Calls `@tauri-apps/plugin-notification` to fire the native OS notification
4. Silent no-op in browsers

## Configuring the Production URL

The current `tauri.conf.json` loads `http://localhost:3100` — the local dev server. Before distributing the app, update the window URL to your Vercel deployment URL:

```json
// src-tauri/tauri.conf.json
{
  "app": {
    "windows": [
      {
        "url": "https://YOUR_VERCEL_URL_HERE"
      }
    ]
  },
  "build": {
    "devUrl": "http://localhost:3100"
  }
}
```

## Development Workflow

**Prerequisites (one-time):**

```
rustup (installed via winget)
cargo (installed with rustup)
```

**Run desktop app in dev mode:**

```bash
# Terminal 1: start the Next.js dev server
npm run dev

# Terminal 2: start the Tauri desktop wrapper
npm run tauri:dev
```

This opens ChefFlow in a native window loading `http://localhost:3100`. Hot reload works — changes to Next.js files update the window automatically.

**Build for distribution:**

```bash
npm run tauri:build
```

Output: `src-tauri/target/release/bundle/`

- `ChefFlow_x.x.x_x64-setup.exe` — installer
- `ChefFlow_x.x.x_x64.msi` — MSI alternative

Note: Initial builds without a code signing certificate will show a Windows SmartScreen warning. Code signing can be added later with a certificate.

## Plugins Used

| Plugin                      | Purpose                                   |
| --------------------------- | ----------------------------------------- |
| `tauri-plugin-autostart`    | Register/unregister Windows startup entry |
| `tauri-plugin-notification` | Send native OS notifications              |

Both are initialized in `main.rs` via `.plugin(...)` and their permissions granted in `capabilities/default.json`.

## What Stays the Same

- All Next.js code — untouched
- Supabase auth — same flows, cookies stored in WebView2 profile
- Stripe payments — same integration
- The website in browsers — works exactly as before

The Tauri shell has no database access, no business logic, and no state of its own. It is purely a window + tray manager.
