# ChefFlow Feature Inventory

This inventory is the current source-of-truth map of implemented ChefFlow functionality discovered from the active codebase, route tree, navigation config, help registry, product audits, schema, and service endpoints.

Project identity, audience, and scope are governed by `docs/project-definition-and-scope.md`. This inventory is implementation-facing and may retain historical naming from older tier or gating systems.

Method used:

- prioritized route families in `app/`, `app/api/`, `components/`, `lib/`, `database/migrations/`, and `clean-schema.sql`
- used route-policy, nav config, help registry, pro-feature registry, and admin flag surfaces as discovery anchors
- treated schema, services, scheduled jobs, and hidden routes as features even when the UI is partial or gated
- normalized overlapping routes into feature families so the registry remains usable as a discoverability layer

Classification rules:

- `chef_standard`: normal chef-facing workflows and the public/client experiences that support them
- `chef_privileged`: advanced owner-level, sensitive, or historically gated chef tools
- `internal_admin`: platform-admin-only operational tooling
- `developer_tool`: engineering, simulation, launcher, or founder-control tooling
- `internal_only`: service-only, background, or non-user-facing systems

Interpretation note:

- `chef_privileged`, `gated`, `requirePro(...)`, and related terms in this inventory describe implementation history, control sensitivity, or complexity. They should not be treated as the canonical public monetization promise unless `docs/project-definition-and-scope.md` is updated to say so.
