# V1 Module Primer

This is the working map Codex should use when deciding where new ChefFlow work belongs.

## Current Truth

The live module taxonomy is generated by:

- `scripts/unified-build-queue/generate.mjs`
- `system/unified-build-queue/candidates.json`
- `system/unified-build-queue/module-batches.json`
- `system/unified-build-queue/summary.json`
- `system/unified-build-queue/approved-batches.json`
- `docs/architecture/v1-module-cards.md`
- `devtools/module-decision.mjs`

This is a build-backlog module map, not yet a complete file-by-file code ownership registry. The generator now carries first-class module, submodule, and assignment fields, but the generated queue artifacts still require review before they become execution truth.

The accepted taxonomy decisions live in:

- `docs/architecture/v1-module-taxonomy-decisions.md`
- `docs/architecture/v1-module-ownership-registry.md`

Those documents are the reviewed architecture baseline. Generated queue artifacts should eventually be updated to emit module and submodule assignments from that baseline.

## Build Rule

Nothing gets built if it does not get moduled.

Before implementation, Codex must identify:

1. Module owner.
2. Submodule owner.
3. Canonical files or route family.
4. Interface the change attaches to or creates.
5. Invariants the module owns.
6. Test surface for the module.

If the work is `unassigned`, cross-module without a named owner, or too fuzzy to map, Codex should route it to module review or planning before code edits.

Non-emergency V1 builds are frozen unless the work maps to an existing module card or a reviewed submodule card. Emergency exceptions are limited to production breakage, security, money, data safety, or explicit Founder Authority override.

## Required Submodule Support

The current generator assigns `module`, `submodule`, and `assignment`. Queue artifacts still need a review pass before high-risk execution uses generated submodule assignments as reviewed truth.

Target shape:

```json
{
  "module": { "id": "auth-security", "label": "Auth and Security" },
  "submodule": { "id": "role-hierarchy", "label": "Role Hierarchy" },
  "assignment": "reviewed"
}
```

Assignment statuses:

- `proposed`: generated by heuristics, not enough for high-risk V1 execution.
- `reviewed`: architecture review confirmed the owner.
- `manual`: explicit assignment because heuristics were wrong or insufficient.
- `unassigned`: not buildable.

V1 blockers and high-risk modules require `reviewed` or `manual` assignment before execution.

## How Codex Should Use This

1. Read the spec or request.
2. Check whether it already has a module in `system/unified-build-queue/candidates.json`, `module-batches.json`, or the V1 builder queue.
3. Read `docs/architecture/v1-module-taxonomy-decisions.md` for accepted submodule policy.
4. Read `docs/architecture/v1-module-cards.md` and `docs/architecture/v1-module-ownership-registry.md` for the selected module.
5. If no queue module exists, infer the likely module and submodule from the accepted taxonomy and verify against code truth.
6. For ambiguous work, run `node devtools/module-decision.mjs --prompt "..."` or `node devtools/module-decision.mjs --source-path path`.
7. Name the module and submodule owner in the spike report.
8. Attach code to that module's current interface or create a narrow interface only when the spec requires it.
9. Reject module theater: do not create shallow files just to satisfy the gate.
10. If the module map is wrong, update the source spec or queue evidence before building.
