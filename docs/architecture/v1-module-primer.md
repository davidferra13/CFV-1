# V1 Module Primer

This is the working map Codex should use when deciding where new ChefFlow work belongs.

## Current Truth

The live module taxonomy is generated by:

- `scripts/unified-build-queue/generate.mjs`
- `system/unified-build-queue/candidates.json`
- `system/unified-build-queue/module-batches.json`
- `system/unified-build-queue/summary.json`
- `system/unified-build-queue/approved-batches.json`

This is a build-backlog module map, not yet a complete file-by-file code ownership registry.

## Build Rule

Nothing gets built if it does not get moduled.

Before implementation, Codex must identify:

1. Module owner.
2. Canonical files or route family.
3. Interface the change attaches to or creates.
4. Invariants the module owns.
5. Test surface for the module.

If the work is `unassigned`, cross-module without a named owner, or too fuzzy to map, Codex should route it to module review or planning before code edits.

## How Codex Should Use This

1. Read the spec or request.
2. Check whether it already has a module in `system/unified-build-queue/candidates.json`, `module-batches.json`, or the V1 builder queue.
3. If no queue module exists, infer the likely module from the table above and verify against code truth.
4. Name the module owner in the spike report.
5. Attach code to that module's current interface or create a narrow interface only when the spec requires it.
6. Reject module theater: do not create shallow files just to satisfy the gate.
7. If the module map is wrong, update the source spec or queue evidence before building.
