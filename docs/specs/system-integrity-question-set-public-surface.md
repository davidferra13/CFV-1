# System Integrity Question Set: Public Surface

> **Purpose:** Expose every failure point in ChefFlow's public-facing surface (60+ pages, 18 token-gated routes, 16 public forms) and force the system into a fully specified, verifiable state.
> **Scope:** Everything under `app/(public)/` including layout, header, footer, all token pages, all forms, all data-display pages, SEO, error handling, caching, and presence tracking.
> **Created:** 2026-04-18
> **Last updated:** 2026-04-18 (initial investigation)
> **Investigator:** Claude Opus 4.6 (main session + 4 parallel haiku investigation agents)
> **Method:** Source code tracing across all 60+ page files, 16 form components, server actions, and shared utilities. No runtime testing.
> **Pre-fix score:** 16.5 / 40 (41.3%)

## A. Token Security & Entropy (4 questions)

| #   | Question                                                                                                                                       | Verdict     | Evidence                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| A1  | Do all token-gated pages use cryptographically strong tokens (>= 128-bit entropy) that resist brute-force enumeration?                         | **FAIL**    | 17 of 18 token pages use strong tokens (128-256 bit: `crypto.randomBytes(32)`, UUID v4, `gen_random_bytes(32)`). **Exception: `/g/[code]`** uses `left(replace(gen_random_uuid()::text, '-', ''), 12)` = 12 hex chars = **48-bit entropy**. Source code comment acknowledges: "short codes are brute-forceable." Rate limit (30 req/15min) mitigates online attacks but 48-bit is weak.                                                                |
| A2  | Do all token pages return `notFound()` or equivalent on invalid/missing tokens, preventing information leakage about valid vs. invalid tokens? | **PASS**    | All 18 token pages return `notFound()` on invalid tokens. `/feedback/[token]` returns a styled "Link Expired" page (softer UX, still no data leak). `/cannabis-invite/[token]` redirects to `/` (feature disabled). No page returns different error shapes for "token exists but expired" vs "token never existed" that would allow enumeration.                                                                                                       |
| A3  | Are token pages protected against unauthorized data modification via token-only access (write operations require additional auth)?             | **PARTIAL** | Most write operations are properly gated: `/cannabis-invite` requires authenticated user, `/tip/[token]` uses CAS guard, `/review/[token]` has one-time submission guard. **But:** `/worksheet/[token]` allows unauthenticated writes that cascade to `clients` and `events` tables with no additional verification. `/hub/join/[groupToken]` allows unauthenticated profile creation. `/share/[token]` allows RSVP updates with cookie-only identity. |
| A4  | Is the `/view/[token]` page rate-limited at the page level, consistent with all other token pages?                                             | **FAIL**    | `/view/[token]` has NO `checkRateLimit` or `enforcePublicActionRateLimit` call at the page level. Every other token page in the system has page-level rate limiting (30-120 req/15min per IP). The data function `resolveViewerInviteContext` increments `view_count` on every load, so an unthrottled caller can inflate view counts and cause unbounded DB writes.                                                                                   |

**Score: 0 PASS, 1 PARTIAL, 2 FAIL** = 1.5 / 4

## E. Rate Limiting & DoS Protection (4 questions)

| #   | Question                                                                                                                          | Verdict     | Evidence                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| --- | --------------------------------------------------------------------------------------------------------------------------------- | ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| E1  | Do all public pages that perform database queries on every request have rate limiting?                                            | **FAIL**    | Data-display pages with DB queries and NO rate limiting: `/chef/[slug]` (multi-table joins), `/chefs` (loads up to 200 chefs + partners), `/ingredients` (paginated but no limit), `/ingredients/[category]`, `/ingredient/[id]` (multi-source lookup), `/nearby` (filtered query), `/nearby/[slug]` (single lookup). Only `/hub/circles` has page-level rate limiting among data pages. An attacker can hammer any data page to stress the DB.                                              |
| E2  | Are unbounded database queries on public pages capped with LIMIT clauses?                                                         | **FAIL**    | `/chef/[slug]`: `getPublicChefReviewFeed` queries 4 tables (`client_reviews`, `chef_feedback`, `external_reviews`, `guest_testimonials`) with **NO LIMIT on any**. `getPublicWorkHistory` and `getPublicAchievements` also have no LIMIT. `/chefs`: loads up to 200 chefs, then bulk-fetches all their partners, marketplace profiles, and directory listings with no per-chef LIMIT. A chef with thousands of reviews would return an unbounded result set.                                 |
| E3  | Does the presence beacon (`PresenceBeacon`) for anonymous users avoid creating a write-amplification DoS vector?                  | **PARTIAL** | Beacon POSTs to `/api/realtime/presence` every 30 seconds per tab, with session ID, pathname, user agent (150 chars), and referrer. Errors are silently caught. The 30s interval is reasonable, but: (1) no rate limiting on the presence API itself is visible from the client code, (2) each tab opens its own beacon, (3) an attacker opening many tabs or scripting POST requests could flood the presence endpoint. Server-side rate limiting on the API route would need verification. |
| E4  | Is the `/chef/[slug]/gift-cards` purchase flow protected against intent-flooding (creating thousands of pending Stripe sessions)? | **FAIL**    | `initiateGiftCardPurchase()` creates a `gift_card_purchase_intents` DB row AND a Stripe Checkout session on every call. No rate limiting. An attacker can automate requests with valid tenant IDs (discoverable from `/chefs` page slugs) to create thousands of pending intents and Stripe sessions, causing DB bloat and potential Stripe rate limit issues.                                                                                                                               |

**Score: 0 PASS, 1 PARTIAL, 3 FAIL** = 0.5 / 4

## Critical Path (Ordered by Blast Radius)

### P0: Security Vulnerabilities

1. **B4: `/nearby/unsubscribe` mass-unsubscribe vulnerability** - Add rate limiting + email ownership verification (confirmation link or logged-in check)
2. **D2: `/hub/join` mass fake profile creation** - Add rate limiting (IP-based, 5 joins/hour) + honeypot
3. **A4: `/view/[token]` missing rate limit** - Add `enforcePublicActionRateLimit` consistent with other token pages

## Verification Protocol

When a builder agent addresses items from the critical path:

1. Read this question set first. Identify which questions your changes affect.
2. After each fix, update the verdict and evidence for the affected question(s).
3. Run `npx tsc --noEmit --skipLibCheck` after changes.
4. Test rate limiting by calling the endpoint rapidly and verifying rejection.
5. Test form validation by submitting empty/malicious input and verifying rejection.
6. Verify OG metadata changes with a link unfurl preview tool.
7. Recompute score. Target: 28/40 (70%) before production marketing push, 32/40 (80%) before scale.
