## Constraints And Guardrails

This research program must remain inside legitimate, defensible bounds.

Allowed:

- public-surface analysis
- archive and third-party intelligence
- legitimate account creation using your own identities and inboxes
- testing normal user flows with your own data
- contacting support through normal channels
- measuring response times and workflow behavior from your own interactions

Not allowed:

- credential abuse
- access-control bypass
- endpoint fuzzing or exploit attempts
- submitting deceptive or harmful production transactions
- creating fake bookings that waste real operator time
- any attempt to access non-public data

If a later step requires real money, a real booking, or a real support burden on a third party, that step needs explicit human approval before execution.

## Exact Execution Order

The next builder should follow this order exactly.

### Phase 0. Preserve the current baseline

Before any new reconnaissance:

1. read the current report and derived-data file
2. verify the saved screenshots and artifacts still exist
3. avoid re-running work that is already represented in the report

Exit condition:

- the builder can clearly distinguish completed baseline research from unresolved gaps

### Phase 1. Finish the highest-value public intelligence

Do this before touching authenticated flows because it is cheap, safe, and improves later interpretation.

Tasks:

- expand Internet Archive work from high-level chronology to route-level diffs
- capture partner validation from partner domains, case studies, and co-marketing pages
- capture third-party review and sentiment surfaces beyond Trustpilot
- capture ad-library and paid-acquisition evidence
- capture broader SEO/traffic-intelligence exports if external tools are available

Required artifacts:

- dated notes with source URLs
- saved screenshots where relevant
- a concise summary of new facts versus previous inferences

Exit condition:

- no major public-facing growth, partnership, or sentiment blind spots remain

### Phase 2. Prepare for legitimate account-based research

Do not start authenticated exploration without the prerequisites.

Prerequisites:

- dedicated research email inboxes
- a clean logging template for every session
- a list of flows to test
- clear rules for what counts as acceptable production interaction

Artifacts to create before exploration:

- account inventory log
- evidence log template
- session-by-session route and behavior tracker

Exit condition:

- the builder has a controlled environment for testing without mixing evidence or causing unnecessary production noise

### Phase 3. Chef-side authenticated product mapping

This is the highest-priority product-insight phase.

Target questions:

- what happens immediately after chef registration
- what the setup and activation funnel actually asks for
- what is required before a chef becomes operational
- what free versus paid surfaces are actually usable
- how inbox, calendar, pricing, quotes, website, integrations, analytics, and upgrade prompts behave
- what hidden or non-marketed features appear after login

Required artifacts:

- full route map of authenticated chef surfaces
- screenshots for every major step
- observed UX flow notes
- upgrade-gate and paywall inventory
- friction / delight inventory

Exit condition:

- a builder can answer how a real chef gets from registration to first productive use without guessing

### Phase 4. Customer-side and transaction-lifecycle validation

Only do this after the chef-side map exists.

Target questions:

- how quote delivery actually works
- what payment and confirmation steps look like
- how cancellation, refund, and complaint flows behave in practice
- how post-service communication and review prompts work
- where the marketplace versus PCM boundary becomes visible to the user

Required artifacts:

- step-by-step flow maps
- screenshots and timestamps
- copies of policy-driven email or UI messages
- evidence of where lifecycle state transitions are visible

Exit condition:

- the report can describe transaction and post-transaction behavior from real observations, not just policy text

### Phase 5. Support and operations validation

Do this with restraint and with normal user behavior.

Target questions:

- how fast support actually replies
- which channels are actually staffed
- whether SLA promises match observed behavior
- how clearly refunds, disputes, and payout issues are explained
- whether support quality differs by plan or issue type

Required artifacts:

- timestamped support log
- channel used
- first-response and resolution timing
- summary of whether the response matched public promises

Exit condition:

- support and service-operation claims are no longer based only on first-party policy pages

### Phase 6. Longitudinal monitoring

This phase should run in parallel after Phases 1 through 3 begin.

Monitor:

- homepage copy changes
- pricing changes
- bundle/hash changes
- legal-page diffs
- help-center diffs
- upgrade copy changes
- review/sentiment drift

Required artifacts:

- dated diff log
- what changed
- why it matters
- whether the change upgrades or weakens a prior inference

Exit condition:

- the research corpus can answer not just what exists now, but what is changing and how quickly

### Phase 7. Synthesis back into product planning

Only after real new evidence exists.

Tasks:

- update `docs/research/competitive-intelligence-takeachef-privatechefmanager-2026-04-02.md`
- add or revise supporting artifact files
- correct `docs/specs/platform-intelligence-hub.md` anywhere the spec still reflects stale assumptions
- clearly separate `Observed`, `Documented`, and `Inferred`

Exit condition:

- the spec and the research corpus no longer drift apart

## Verification Rules

Every new claim should be tagged one of three ways:

- `Observed`
- `Documented`
- `Inferred`

Every major conclusion should answer:

1. what was directly seen
2. where it was seen
3. what remains unverified

Do not collapse inference into fact.

## Common Failure Modes

The next builder is likely to get these wrong if the work is rushed:

- redoing public-surface work instead of closing operator-level gaps
- mixing present-tense facts with old archived states
- turning marketing copy into assumed shipped functionality
- mistaking help-center promises for observed operational behavior
- using one logged-in surface to generalize the whole product
- failing to record timestamps, which makes pricing/support claims decay quickly
- updating the spec without updating the evidence base
