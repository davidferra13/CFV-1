# Web Beta Release Architecture

Status date: `2026-03-16`

## Decision

ChefFlow should ship as a hosted invite-only web beta first.

Do not treat source checkout, desktop installers, or app-store packages as the release path for external users. The exact build going forward is:

1. Build a production Next.js artifact.
2. Deploy that artifact to the beta host.
3. Distribute only the hosted beta URL.
4. Add PWA install only after update behavior is proven.
5. Defer native desktop and mobile until they have a real updater, signing, and rollback model.

## Architecture Specification

### Runtime Surfaces

- Primary runtime: Next.js App Router hosted web app.
- Public distribution surface: `https://beta.cheflowhq.com`
- Public launch surface later: `https://app.cheflowhq.com`
- Fast liveness: `/api/health/ping`
- Public health contract: `/api/health`
- Strict release readiness: `/api/health/readiness?strict=1`

### Build Artifact

- Artifact type: production Next.js build output generated by `npm run build`
- Required build mode: production, never `next dev`
- Dist isolation: unique `NEXT_DIST_DIR` per verification run to avoid dev/build contamination
- Release verification: `npm run verify:release:web-beta`

### Code Boundaries

- Client-triggered mutations live in dedicated `use server` action modules.
- Server-rendered reads live in dedicated query modules instead of mixed action/query files.
- Public health behavior is implemented from one shared core so `/api/health` and `/api/health/readiness` cannot drift independently.
- Release-smoke Playwright config must boot `next start`, not `next dev`.

### Infrastructure

- Release host for beta: Developer PC (localhost:3200) behind Cloudflare Tunnel
- Public production host later: Vercel
- Required environment isolation:
  - beta URL vars
  - PostgreSQL redirect URLs
  - Google OAuth callbacks
  - Stripe mode separation
  - `CRON_SECRET`

### Monitoring

- Liveness monitor targets `/api/health/ping`
- Paging monitor targets `/api/health/readiness?strict=1`
- Sentry owns errors and traces
- PostHog owns activation and retention telemetry

### Distribution Model

- External users get the hosted beta URL only
- Optional convenience install later: browser-based PWA install
- Explicitly deferred:
  - Tauri desktop binaries
  - App Store / Play Store distribution
  - self-hosted customer installs

## Release Gate Contract

Every beta candidate must pass:

1. `npm run build`
2. `npm run test:unit`
3. `npm run verify:release:web-beta`

`verify:release:web-beta` is expected to execute:

1. secrets scan
2. beta typecheck
3. beta lint
4. critical test suite
5. beta release unit contract tests
6. beta production build
7. beta production smoke test

## Implementation Timeline

### Phase 0: Release Contract Lock

Target: `1 day`

- Freeze the hosted web-beta release path.
- Split server actions from server-only reads where the same module was serving both.
- Unify public health logic behind one implementation.
- Ensure beta smoke runs against `next start`.

### Phase 1: Build Stabilization

Target: `2-4 days`

- Get `npm run build` green on the release candidate branch.
- Eliminate build-time server boundary violations.
- Fix remaining release-path blockers in auth, health, and feedback flows.
- Make `npm run verify:release:web-beta` pass end to end.

### Phase 2: Operational Hardening

Target: `2-3 days`

- Verify `deploy-beta` and `rollback-beta` against the green release candidate.
- Point uptime monitors to strict readiness.
- Confirm Sentry and PostHog environment separation.
- Perform one rollback drill.

### Phase 3: Controlled External Beta

Target: `1-2 weeks`

- Recruit 5 initial chefs.
- Run white-glove onboarding.
- Verify first-use activation and feedback capture.
- Expand only after the first cohort completes core workflows without manual rescue.

### Phase 4: Broader Launch Readiness

Target: `after beta evidence exists`

- Decide whether PWA install improves activation.
- Do not start native desktop/mobile packaging until update and rollback mechanics are productionized.

## Non-Negotiable Exit Criteria

- Production build is reproducible.
- Release gate is green on the exact deployed commit.
- Rollback is documented and recently tested.
- Health monitors page on real degradation.
- External users do not depend on local setup, source checkout, or unstable native wrappers.
